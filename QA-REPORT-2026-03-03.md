## Shrnuti

System Memoriq ma **solidni zaklady** — WAL rezim, 30s busy_timeout, spravny cleanup spojeni, graceful degradation pri zamceni. Audit vsak odhalil **4 kriticke problemy**, **9 zavaznych problemu** a **12 strednich problemu**, ktere dohromady predstavuji vyznamne riziko pro ztratu dat, mezery v kontextu a spolehlivost pri soubehem zatezi.

Tri nejnalejhavejsi problemy:
1. **39 % sessions je osirelych** a crash recovery je nemuze opravit (stale session soubory ji blokuji)
2. **Auto-linking NIKDY nefungoval** v produkci (0 linku pres 189 faktu — threshold je prilis prisny)
3. **Rozbite importy** v Codex CLI integraci a AGENTS.md generatoru (ImportError pri volani)

**Statistiky DB v dobe auditu:**
- 79 sessions: 48 uzavrenych, **31 osirelych** (21 zablokovanych od recovery)
- 189 faktu: **156 bez source_file** (zadna ochrana proti duplikatum)
- 0 fact_links, 0 causal_chains, 0 clusters — grafove funkce nevyuzity
- 78 code souboru, 551 symbolu, 3466 referenci zaindexovano
- 21 projektu, 187 vektorovych embeddingu

## KRITICKE Problemy (4)

### C1: Rozbite Importy — session_init.py a generate_agents_md.py

Oba soubory importuji `check_crash_recovery` a `auto_detect_identity` z `on_session_start.py`, ale tyto funkce byly pri optimalizaci vykonu presunute do `tools/project_context.py`. **Volani kterehokoliv souboru spadne na ImportError.**

- `generate_agents_md` = generator AGENTS.md — kompletne nefunkcni

**Dopad:** Integrace Codex CLI je nefunkcni.

**Oprava:** Aktualizovat importy na `tools.project_context` nebo re-exportovat z `on_session_start.py`.

### C2: 31/79 Sessions Osirelych, Crash Recovery Zablokovana

**Soubor:** `tools/project_context.py:71-75`

Crash recovery preskoci kazdou osirelou session, jejiz per-session soubor (`~/.memoriq/sessions/{claude_sid}.json`) existuje:

```python
if session_file.exists():
    continue  # Session stale ziva, preskocit
```

**Realita:** 21 z 31 osirelych sessions ma session soubory na disku. Tyto soubory se nikdy neuklizi, protoze `cleanup_old_sessions()` bezi pouze uvnitr `on_session_end.py` a SessionEnd se spusti jen v ~11 % pripadu.

**Retezec korenovych pricin:**
1. Claude provede context compacting → SessionStart se spusti znovu se stejnym `claude_session_id`
2. Vytvori se novy radek session, stary zustane otevreny (zadne `UPDATE ... SET end_time` pro predchozi)
3. Per-session soubor se prepise (stejny nazev), ale v DB jsou ted 2 otevrene sessions pro dane ID
4. SessionEnd se pro starou session nikdy nespusti → permanentni sirotek
5. Session soubor existuje → crash recovery ho navzdy preskoci

**Dukaz:** 4 hodnoty `claude_session_id` maji vice osirelych sessions (2-3 kazda).

**Oprava (vice kroku):**
1. V `create_session()`: uzavrit jakoukoli predchozi session se stejnym `claude_session_id` pred vytvorenim nove
2. V crash recovery: pridat casovy cleanup (uzavrit sessions starsi nez N hodin bez ohledu na soubor)
3. Pridat pravidelny cleanup session souboru (smazat soubory pro uzavrene sessions)

### C3: Race Condition v Memory Write pri Deduplikaci

**Soubor:** `tools/memory_write.py:137-163`

Deduplikacni cesta provadi vzor cteni-zmena-zapis BEZ izolace transakci:

```python
existing = db.execute("SELECT ... WHERE project=? AND source_file=? AND type=?", ...)
if existing:
    if existing[1] == content:
        return  # beze zmeny
    db.execute("UPDATE facts SET content=? ... WHERE id=?", ...)
```

Pri 3-5 CLI instancich mohou dve precist stejny radek `existing` soucasne. Druhy UPDATE tise prepise prvni. WAL rezim serializuje zapisy, ale druhy zapisovatel pouziva zastarala data ze sveho cteni.

**Dopad:** Ovlivnuje pouze fakty s nastavenym `source_file` (aktualne 33 z 189), ale toto je "chytra" deduplikacni cesta, ktera ma byt bezpecna.

**Oprava:** Pouzit `BEGIN IMMEDIATE` nebo `INSERT ... ON CONFLICT DO UPDATE` (UPSERT) pro atomicitu.

### C4: Auto-Linking Nikdy Nefungoval — 0 Linku Pres 189 Faktu

**Soubor:** `tools/memory_write.py:51-52`

Threshold pro auto-link je nastaven na `distance > 0.25` (kosinova vzdalenost). Testovani ukazuje, ze i uzce souvisejici fakty maji vzdalenosti 0.67+. Threshold je prilis prisny — vyzadoval by temer identicky text.

**Dukazy:**
- tabulka `fact_links`: **0 radku** (pres vsech 189 faktu, vsech 21 projektu)
- tabulka `causal_chains`: **0 radku**
- Fakty s `cluster_id`: **0**
- Vektorove hledani SQLite faktu vuci jinemu SQLite faktu: vzdalenost = 0.67

Cela vrstva znalostniho grafu (linky, retezce, clustery, kontradikce) je efektivne **mrtvy kod**. Konsolidace bezi, ale nenajde nic ke clusterovani, protoze nejsou zadne linky.

**Oprava:** Zvysit threshold na `distance > 0.6` (podobnost > 0.4) nebo lepe — analyzovat skutecnou distribuci vzdalenosti a nastavit threshold na 80. percentil intra-projektovych vzdalenosti.

## ZAVAZNE Problemy (9)

### H1: Zadna Obsahova Deduplikace pro Fakty Bez source_file

**Soubor:** `tools/memory_write.py:137`

Deduplikace funguje pouze pro fakty s nastavenym `source_file A type`. **156 z 189 faktu (82,5 %)** nema `source_file` a nema ZADNOU ochranu proti duplikatum.

Pokud Claude zapise stejny fakt dvakrat (bezne behem dlouhych sessions nebo po context compacting), obe kopie se ulozi. Aktualne 0 obsahovych duplikatu existuje, ale jen proto, ze pouzivani bylo lehke. S rustem poctu projektu a sessions se duplikaty budou hromadit.

**Oprava:** Pridat FTS5 kontrolu podobnosti obsahu: pred vlozenim vyhledat existujici fakty s velmi podobnym obsahem ve stejnem projektu+typu.

### H2: Duplicitni Sessions pri Context Compacting

**Soubor:** `hooks/on_session_start.py:270`

`create_session()` vzdy vlozi novy radek bez kontroly existujicich otevrenych sessions se stejnym `claude_session_id`. Context compacting spusti SessionStart znovu, coz vytvori duplicitni sessions. Toto je korenova pricina C2.

**Oprava:** Pridat `UPDATE sessions SET end_time = ? WHERE claude_session_id = ? AND end_time IS NULL` pred `create_session()`.

### H3: Nekonzistence Bridge Dotazu Pres 3 Komponenty

Tri komponenty nacitaji posledni bridge s ruznym SQL:

| Komponenta | Filtr | Vysledek |
|-----------|--------|--------|
| `on_session_start.py:148` | `end_time IS NOT NULL` | Pouze dokoncene sessions |
| `session_bridge.py:22` | Zadny filtr | Jakakoliv session vcetne osirelych |
| `project_context.py:221` | Zadny filtr | Jakakoliv session vcetne osirelych |

**Dopad:** Behem startu CLAUDE.md dostane bridge z posledni dokoncene session. Ale `session_bridge("load")` muze vratit zastaraly bridge z osirele session.

**Oprava:** Sjednotit vsechny 3 dotazy na stejny filtr (prednostne `end_time IS NOT NULL`).

### H4: on_file_change.py Pouziva Surove sqlite3.connect, Nizky busy_timeout

**Soubor:** `hooks/on_file_change.py:81-82`

Tento hook otvira surove `sqlite3.connect()` pouze s `busy_timeout=2000` (2 sekundy) oproti 30 sekundam vsude jinde. Pri tezkem soubehem zatezi hook selze pri zapisu zaznamu zmen. Selhani jsou logovana pouze na urovni DEBUG — kompletne neviditelna.

**Dopad:** Tise ztraci data o sledovani zmen souboru. Nouzove bridge postavene z tabulky `changes` budou nekompletni.

**Oprava:** Pouzit `open_db_fast()` z `db.py` pro konzistentni nastaveni. Nebo minimalne zvysit na `busy_timeout=10000` a logovat na urovni WARNING.

### H5: Hromadne UPDATy Heat Decay Blokuji Vsechny Zapisovatele

**Soubor:** `tools/memory_search.py:74-105`

Heat decay nacte VSECHNY fakty pro projekt a aktualizuje je jeden po druhem v jedne transakci. Pro projekty se 100+ fakty to drzi zapisovy zamek po vice sekund.

**Horsi:** Kazda MCP server instance ma svuj vlastni `_last_decay_times` slovnik (radek 40). Kdyz 5 CLI vsechny hledaji ve stejnem projektu poprve, VSECH 5 nezavisle spusti heat decay soucasne, coz zpusobi 5 soutezicich hromadnych transakci.

**Oprava:**
1. Ulozit cas posledniho decay do DB (sdileny pres instance)
2. Pouzit jeden SQL UPDATE s CASE vyrazem misto per-radkovych updatu

### H6: Zadne Automaticke Ulozeni Bridge Pred Context Compacting

**Architekturalni problem.** Kdyz Claude provede compacting kontextu, neexistuje hook pro ulozeni stavu. LLM ma pokyn ukladat bridge po ukolech (Tier 1 v CLAUDE.md), ale:
- Context compacting spousti Claude Code, ne LLM
- Pokud nebyl bridge nedavno ulozen, rozdelany kontext se ztrati
- Nouzovy bridge (ze zmen v DB) zachytava pouze Write/Edit operace, ne praci v Bashi ani uvazovani

**Castecne zmirnovani existuje:** PostToolUse hook sleduje zmeny souboru. Ale chybi vsechny zmeny provedene pres Bash.

### H7: Konsolidace Blokuje Vsechny Zapisovatele Behem 4 Tezkych Transakci

**Soubor:** `tools/consolidate.py:294-310`

Konsolidace bezi 4 sekvencni tezke transakce (`_find_clusters`, `_compute_tiers`, `_detect_contradictions`, `_summarize_clusters`), kazda s mnoha ctenimi a zapisy. Behem teto doby ostatni CLI instance zaznamenavaji zpozdeni zapisu.

**Dopad:** Problem spolehlivosti/odezvy, ne ztrata dat. Ale mohlo by zpusobit timeouty tool callu.

**Oprava:** Pridat sledovani postupu a yield mezi transakcemi. Zvazit spousteni konsolidace pouze kdyz je system v klidu.

### H8: Castecne Indexovani Kodu Ponechava Reference Trvale Nevyresene

**Soubor:** `code/indexer.py:144-191`

Pokud je `code_index` prerusen casovym budgetem, commituje per-soubor. `resolve_references` bezi na castecne sade symbolu, coz ponechava mnoho referenci nevyresenych. Soubory indexovane v prvnim castecnem behu nebudou "dirty" pro dalsi inkrementalni index, takze reference se nikdy znovu nevyresi.

**Oprava:** Oznacit soubory jako "needs_resolve" a re-resolvovat v dalsim indexovacim behu, nebo spustit samostatny resolve pruchod.

### H9: Zadna Tabulka Sledovani Nacitani / Zadna Tabulka Verze Schematu

**Dukaz z DB:** tabulka `retrieval_log` existuje, ale ma 0 sloupcu (prazdne schema). Tabulka `schema_version` neexistuje.

**Dopad:**
- Nelze sledovat, ktere fakty jsou skutecne uzitecne (heat boosting na zaklade nacitani je prazdny)
- Nelze overit, zda je schema aktualni, nebo sledovat migrace

## Diskuzni Bod: Skalovani Databaze (WAL2 / Replikace)

Uzivatel nastolil myslenu replikace databaze pro lepsi soubehy zapis. Analyza:

**Aktualni stav:** SQLite WAL rezim umoznuje JEDNOHO zapisovatele najednou. Pri 3-5 CLI se zapisovatele radi za sebe (30s timeout). Toto funguje, ale vytvari spicky latence behem tezkych zapisovych operaci (heat decay, konsolidace, indexovani kodu).

**Moznosti k zvazeni (budoucnost):**
1. **SQLite WAL2 rezim** (experimentalni) — umoznuje soubehy zapisovatele do ruznych WAL souboru
2. **Write-Ahead Buffer** — frontat zapisy v pameti/souboru, davkove commitovat periodicky
3. **Per-projektove DB shardovani** — kazdy projekt dostane svuj vlastni .db soubor, eliminuje cross-projektovou kontenci
4. **LiteFS / Litestream** — pro replikaci (ale single-writer, nepomaha soubehu)

**Doporuceni:** Per-projektove DB shardovani je nejpraktictejsi vylepseni. Fakty, sessions a zmeny kazdeho projektu jdou do `~/.memoriq/projects/{nazev}/memory.db`. Sdilene tabulky (projects, identity, tech_templates) zustanou v hlavni DB. Toto kompletne eliminuje kontenci mezi ruznymi projekty.

**Ale nejdriv:** Opravit kriticke problemy vyse. Stavajici architektura s jednou DB je pro 3-5 CLI v poradku, pokud budou zapisove operace optimalizovany (davkovy heat decay, koordinace pres instance).
